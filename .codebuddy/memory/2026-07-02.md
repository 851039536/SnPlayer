## 修复并行加密/解密版本字节截断 bug 并启用多线程加密

### 根因
`crypto_service.dart` 中 `_writeBatchToOutput` 使用 `FileMode.write`（等同 `O_TRUNC`）打开输出文件。并行加密流程第 5 步先写入 64 字节文件头（含版本号 0x02），关闭文件；随后 `_writeBatchToOutput` 重新用 `FileMode.write` 打开 → **截断！文件头清零，版本字节变 0x00**。此外 `parallelDecryptMaxConcurrency=2`，当 isolateCount>2（256MB+ 文件）时 `_writeBatchToOutput` 被多次调用，每次截断导致前批 chunk 数据也丢失（解密同样受影响，只是之前未暴露）。

### 修复（`lib/services/crypto_service.dart`，3 处改动）
1. **`_writeBatchToOutput`** — 签名改为接收 `RandomAccessFile outputRaf`（已打开句柄），移除内部 `openSync`/`closeSync`，变为纯写入逻辑，生命周期由调用方管理
2. **`_runParallelEncrypt`** — 第 5 步写 header + 预分配后**保持句柄打开**，各批次复用同一句柄调 `_writeBatchToOutput(outputRaf, ...)`，`try/finally` 保证异常时关闭
3. **`_runParallelDecrypt`** — 同上，第 4 步预分配后保持句柄打开，`try/finally` 保护

### `encryptFile` 自动选路
`encryptFile` 已是自动选择路径：>= `parallelDecryptMinFileSize`（64MB）走 `encryptFileParallel`，否则串行。调用方 `video_list_provider.dart` 零改动，透明升级。

### 文档更新
`CODEBUDDY.md` 移除"并行加密代码保留但不应被启用"的警告，更新为已修复并自动启用说明。

## 播放解密代码审查修复

### 高优内存泄露修复
1. **`_initWithProxy` _controller 泄露**（`video_player_screen.dart`）：`_initWithProxy` 内部 `initialize()`/`play()` 抛异常时 `_controller` 未 dispose，导致 ExoPlayer 原生资源泄露。修复：内部加 try-catch dispose + rethrow，外层 `_initPlayer` 阶段 2 catch 补充 dispose。
2. **worker cancelled 全局竞态**（`streaming_decrypt_proxy.dart`）：全局 `bool cancelled` 被所有 decrypt_range 共享，并发 Range 请求（视频轨+音频轨）互相干扰——请求 B 到达时 `cancelled=false` 会"取消取消"请求 A，seek 时 `cancelled=true` 误取消所有请求。修复：改为 `int? cancelledRequestId` 匹配机制，每个请求分配唯一 ID，cancel 携带目标 ID，`isCancelled` 检查 `cancelledRequestId == requestId`。

### 中优修复
3. **`_playExternal` NO_PLAYER 分支泄露**（`video_list_screen.dart`）：openUrl 抛 NO_PLAYER 时直接 return，`_externalProxy` 未清理。修复：return 前 `await _externalProxy?.stop(); _externalProxy = null;`。
4. **`stop()` 冗余 send**（`streaming_decrypt_proxy.dart`）：`send('stop')` 后立即 `kill(immediate)`，消息不被处理。移除冗余 send。
5. **dispose unawaited**（两个 screen）：`_proxy.stop()` / `_externalProxy.stop()` 改为 `unawaited(...)` 明确 fire-and-forget 意图。

### 文件变更

- `lib/screens/video_player_screen.dart` — _controller 泄露修复 + unawaited

## 修复流式解密代理 seek 回退黑屏 + HttpException

### 用户反馈
1. `HttpException: Content size below specified contentLength. 2097152 bytes written but expected 4472155`（4 个 512KB 块后连接提前关闭）
2. seek 从左往右（前进）正常，拉回左边（回退）有时黑屏后异常

### 根因（4 个）
1. **`_handleRequest` close() 抛 HttpException**：seek 导致旧连接断开，`_decryptAndStream` 提前终止但 `_handleRequest` 仍调 `close()`，实际写入 < contentLength 头声明值 → `HttpException` + 连接异常关闭 → 播放器收到不完整响应 → 黑屏
2. **阶段 1 缓存命中无 try-catch**（338-360行）：seek 回退时缓存命中块写入已断开的连接，异常直接传播到外层 catch，未发 cancel 给 worker
3. **worker `cancelledRequestId` 单一变量覆盖**（507行）：视频轨+音频轨并发 cancel 时，后发 cancel 覆盖前一个，旧请求无法取消 → worker 任务泄漏
4. **worker ack 等待死锁**（656行 `await ackReceivePort.first`）：主线程在收到第一个 block 前断开（ackPort=null），worker 发完 2 块后永远卡在 await ack → encFile 泄漏 + worker 资源被占用

### 修复（`lib/services/streaming_decrypt_proxy.dart`，5 处改动）
1. **`_handleRequest`** — `_decryptAndStream` 返回 `bool`（是否完整写入）。完整写入 → 正常 `close()`；提前终止 → `detachSocket().destroy()` 重置 TCP 连接（发 RST 让播放器明确知道中断），fallback 到 close + 忽略异常
2. **`_decryptAndStream` 阶段 1** — `response.add/flush` 包裹 try-catch，连接断开时 `return false`
3. **`_decryptAndStream` 阶段 2** — 所有 `break` 改为 `return true/false`：`done` → true（完整），`cancelled`/`error`/连接断开 → false
4. **`_decryptWorkerEntry`** — `int? cancelledRequestId` → `Set<int> cancelledRequests`（支持并发取消）；新增 `Map<int, SendPort> requestAckPorts`；cancel 回调中 `add(rid)` + `requestAckPorts[rid]?.send('cancel-ack')` 唤醒卡在 await ack 的任务；finally 中清理 ackPort 和取消标记
5. **`_decryptRangeInWorker`** — 新增 `requestId` + `requestAckPorts` 参数，开始时 `requestAckPorts[requestId] = ackReceivePort.sendPort` 注册

## 修复 worker "Stream has already been listened to" 错误

### 用户反馈日志
- `worker 错误: Bad state: Stream has already been listened to.` 反复出现
- `ExoPlayer: ParserException: Invalid NAL length` 解码失败
- seek 后视频渲染从 smooth 变 not smooth，最终播放器报 Source error

### 根因
`_decryptRangeInWorker` 中 `await ackReceivePort.first` 对同一个 `ReceivePort` 多次调用。
`ReceivePort` 是 single-subscription stream，`first` 内部调 `listen(cancelOnError: true)`，
取消订阅时端口被关闭。第二次 `first` 再 `listen` → 抛 "Stream has already been listened to"。

worker 异常退出 → 解密中断 → 后续请求拿到不完整/空数据 → ExoPlayer `Invalid NAL length` → 播放错误。

### 修复（`streaming_decrypt_proxy.dart`，3 处改动）
1. 引入 `StreamIterator(ackReceivePort)` 替代 `ackReceivePort.first`
2. `await ackReceivePort.first` → `await ackIterator.moveNext()` — StreamIterator 保持单个持久订阅，可多次 moveNext
3. finally 块：`await ackIterator.cancel()` 后再 `ackReceivePort.close()`

## 修复缓存块索引错位导致 Invalid NAL length（第四轮）

### 用户反馈日志
- `Stream has already been listened to` 已消失（上轮修复生效）
- 但 seek 多次后仍有 `ExoPlayer: ParserException: Invalid NAL length` → 播放器报 Source error
- 大量 `pipelineFull: too many frames in pipeline (6)` + render smooth/not_smooth 反复抖动

### 根因
worker 中 `isFullBlock = skipBytes == 0 && bytesRead >= blockSize`，`blockIndex = alignedStart ~/ blockSize`。
当 rangeStart 是 16 字节对齐但不是 512KB 对齐时（如 rangeStart=96），alignedStart=96，skipBytes=0，isFullBlock=true，
缓存到 blockIndex=0。但缓存的数据是明文 96-524384，不是块 0 应有的 0-524287。

后续 seek 回退到位置 0 时，缓存命中 blockIndex=0，返回错位数据（前 96 字节是明文 96-191 而非 0-95）→
MP4 NAL 长度字段错误 → ExoPlayer `Invalid NAL length` → 播放器崩溃。

### 修复（`streaming_decrypt_proxy.dart`，1 处改动）
`isFullBlock` 增加 `alignedStart % blockSize == 0` 条件，只有 512KB 对齐的块才缓存。
