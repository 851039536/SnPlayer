## SnPlayer 流式解密代理优化

为 SnPlayer Flutter 项目实现了本地 HTTP 代理流式解密方案，替代原有"全量解密后播放"模式，解决大文件解密等待过长问题。

### 新增文件
- `android/app/src/main/res/xml/network_security_config.xml` — Android 网络安全配置，允许 127.0.0.1/localhost 明文 HTTP
- `lib/services/streaming_decrypt_proxy.dart` — 本地 HTTP 代理核心（HttpServer + Range 解析 + AES-CTR 随机访问解密 + 256KB 块流式响应 + 内存 LRU 块缓存 + 磁盘缓存写入）
- `lib/services/playback_cache_manager.dart` — 磁盘缓存管理（完整性校验 + 路径生成 + 过期清理 + LRU 总量管理）

### 修改文件
- `android/app/src/main/AndroidManifest.xml` — 添加 `android:networkSecurityConfig` 引用
- `lib/config/crypto.dart` — 新增流式解密常量（streamingChunkSize=256KB, streamingBlockSize=4MB, streamingMaxCacheBlocks=16, playCacheMaxSize=500MB, playCacheExpireDays=3 等）
- `lib/services/crypto_service.dart` — 新增 `getEncryptedFileInfo()` 方法 + `EncryptedFileInfo` 数据类
- `lib/screens/video_player_screen.dart` — 三段式播放流程（缓存检查→代理播放→降级全量解密）

### 核心架构
利用 AES-256-CTR 随机访问特性 + video_player 的 networkUrl() + 原生播放器 HTTP Range 支持：
1. 播放器发送 Range 请求 → 代理计算密文偏移 + 调整 IV counter
2. 仅解密请求区间的 256KB 块 → 流式返回 + 写入磁盘缓存
3. 二次播放直接从磁盘缓存加载，零解密等待
4. 代理异常自动降级到原有全量解密路径

### 关键技术点
- CTR 模式 16 字节对齐：`aligned_start = (range_start ~/ 16) * 16`，跳过多余字节
- 复用已有 `CryptoUtils.incrementCounter()` 和 `CryptoUtils.createCtrCipher()`
- 内存块缓存 LRU（64MB 上限），磁盘缓存按偏移写入 + 大小校验
- 零新依赖（使用 dart:io HttpServer）

## 流式代理并发竞态 + Content-Type 修复

用户反馈：个别视频播放失败（`contentIsMalformed`）；暂停后卡死无法继续播放。

### 根因分析
1. **Content-Type 固定 `video/mp4`**：非 MP4 格式（MKV/AVI/FLV 等）被 ExoPlayer 用 MP4 解析器解析，导致 `contentIsMalformed`
2. **共享 `RandomAccessFile` 并发竞态**：ExoPlayer 并发发送多个 Range 请求（文件头 + moov atom），多个请求共享同一个 `_encFile`，在 `await` 点交错执行 `setPosition`，导致读到错误位置的数据 → 解码器卡死

### 修复（`streaming_decrypt_proxy.dart` 重写）
1. **动态 Content-Type**：新增 `_guessContentType()`，从加密文件名提取原始扩展名，映射到对应 MIME 类型（mp4/mkv/avi/mov/flv/wmv/webm/3gp/ts）
2. **每请求独立文件句柄**：`_decryptAndStream()` 内部用 `File(_encPath).open()` 打开独立句柄，`finally` 中关闭，彻底消除并发 `setPosition` 竞态
3. **磁盘缓存写入互斥锁**：新增 `_Mutex` 类（基于 `Completer` 队列），`_writeToCache()` 通过 `synchronized()` 串行化写入，避免缓存文件 `setPosition` 竞态

## 流式代理供数速度优化（解决播放卡顿）

用户反馈：播放 2 秒卡住暂停，点播放又 2 秒卡住，画面卡卡的。

### 根因分析
1. **磁盘缓存写入阻塞响应流**：每 256KB 块解密后 `await _writeToCache()` 串行等待磁盘 I/O，阻塞了给 ExoPlayer 的供数
2. **事件循环让出过于频繁**：每 256KB `Future.delayed(Duration.zero)` 一次，调度开销碎片化吞吐量
3. **处理块太小**：256KB 块循环次数多，系统调用开销大

### 修复
1. **处理块从 256KB → 1MB**：新增 `_decryptBlockSize = 1MB` 常量，减少循环和系统调用次数 4 倍
2. **磁盘缓存写入改为 fire-and-forget**：`unawaited(_writeToCache(...))`，复制数据后异步写入，不阻塞 HTTP 响应流
3. **事件循环让出频率降低**：从每 4MB 改为每 8MB 让出一次
4. **磁盘缓存写入完全移除**：流式传输期间不再写磁盘缓存（fire-and-forget 产生 1-2MB 内存复制 + 事件循环积压，是最大瓶颈），`_writeToCache`/`_cacheMutex`/`_Mutex` 类全部移除
5. **块大小 1MB→2MB**：减少循环和系统调用次数

## 700MB 大视频播放 21 秒后卡死修复

用户反馈：700MB 视频播放前 21 秒正常，之后卡死。日志显示 `pipelineFull: too many frames (6)` 持续数百次、`AudioTrack pause` 在 20s 触发、音频解码器 `Render:0, Drop:236`（全部丢弃）。

### 根因
`response.add()` 是 fire-and-forget（Dart IOSink.add 返回 void，非 Future）。代理解密数据后直接入队不等待发送，700MB 文件全量排队内存导致：
1. 内存累积 → GC 压力 → 事件循环暂停
2. 突发数据灌入 → ExoPlayer pipelineFull（管道容量仅 6 帧）
3. 音频轨 Range 请求被阻塞 → AudioTrack 供数不足 → 20s 暂停 → 卡死

此外 2MB 块 + 16MB 让出间隔 = ~400ms 同步阻塞，并发 Range 请求无法及时处理。

### 修复（streaming_decrypt_proxy.dart + crypto.dart）
1. **每块 `await response.flush()`** — 提供背压，等数据写入 socket 再解密下一块，避免全量排队
2. **`_decryptBlockSize` 2MB → 512KB** — 单块同步解密从 ~100ms 降至 ~25ms
3. **每块 `await Future.delayed(Duration.zero)`** — 替代旧的 16MB 让出逻辑，并发 Range 请求及时响应
4. **块缓存粒度 4MB → 512KB** — 匹配新块大小（旧条件 `bytesRead >= streamingBlockSize` 在 512KB 块下永不成立）
5. **`streamingMaxCacheBlocks` 16 → 128** — 维持 64MB 总缓存（128 × 512KB）
6. 移除 `_initialBurstSize`/`_yieldInterval`/`_shouldYield` — flush + 每块让出已覆盖

用户反馈：上轮修复（512KB+flush+Duration.zero）后 700MB 视频仍 pipelineFull、音频全部丢弃。

## 节流参数调优 + 缓存命中路径去节流（解决 Source error）

用户反馈：上轮节流修复后部分视频播放报 `ExoPlaybackException: Source error`。

### 根因
节流逻辑对**缓存命中路径**也生效，且 burst 仅 2MB 太小。ExoPlayer seek 后或音频轨并发 Range 请求命中缓存时仍被 20ms/块延迟拖慢，累积延迟可能触发 ExoPlayer 内部超时 → Source error。不同视频码率/格式对供数延迟敏感度不同，所以只有"部分视频"出错。

### 修复（crypto.dart + streaming_decrypt_proxy.dart）
1. **`crypto.dart` 参数调优**：
   - `streamingThrottleDelayMs` 20 → 15（有效吞吐 ~34MB/s，仍远超视频消费）
   - `streamingBurstBlocks` 4 → 16（前 8MB 免延迟，覆盖绝大多数 Range 请求）
2. **`streaming_decrypt_proxy.dart`**：
   - 缓存命中路径移除节流（内存数据不会溢出管道，立即返回）
   - 仅解密路径保留节流逻辑（防 pipelineFull）
   - `blocksSent` 计数器仅用于解密路径

## 流式解密播放卡顿优化（cipher 复用 + 移除节流）

用户反馈：播放有点卡顿，跟掉帧一样，特别是大文件。

### 根因分析（streaming_decrypt_proxy.dart `_decryptAndStream`）
1. **每块重建 cipher（最严重）**：每个 512KB 块都 `incrementCounter` + `createCtrCipher`。700MB ≈ 1400 次对象创建+GC。CTR 流式 cipher 连续 `processBytes` 自动递增 counter，顺序播放无需重建。
2. **每块 flush + 15ms 人为节流**：burst 16 块后每块额外 15ms。有效供数 ≈ 10MB/s，高码率/seek 后缓冲重建供不上。flush 本身已提供 TCP 背压，delay 冗余。
3. **每块冗余 setPosition**：顺序播放时文件句柄位置自然前进，仍每次 setPosition。

### 修复（2 个文件）
1. **streaming_decrypt_proxy.dart**：
   - 新增 `StreamCipher? cipher` + `cipherPos` 跟踪变量，仅位置不连续时重建（seek/首次/缓存命中后）
   - 新增 `lastFilePos` 跟踪文件句柄位置，仅不匹配时 setPosition
   - 移除 `blocksSent` 计数器和 `Future.delayed(throttleDelay)` 节流逻辑，仅保留 `flush()` 背压
   - 新增 `import 'package:pointycastle/api.dart'` 获取 `StreamCipher` 类型
2. **crypto.dart**：移除未使用的 `streamingThrottleDelayMs` 和 `streamingBurstBlocks` 常量

### 性能预期
供数速率从 ~10MB/s 提升至 ~20-25MB/s（2-2.5x），消除对象创建/GC/setPosition 开销。lint 0 error。

### 修正：移除节流导致更慢（"全量解密"现象）

用户反馈：移除节流后更慢了，像全量解密。

**根因**：`flush()` 只等数据进入 OS socket 缓冲区，不等播放器消费。移除 `Future.delayed` 后：
1. 解密器跑在播放器前面，超前解密大量不需要的数据（"全量解密"现象）
2. 同步 `processBytes`（25ms/块）密集阻塞事件循环，并发 Range 请求处理延迟（"更慢"）

**修复**：恢复 burst + 节流机制，但延迟从 15ms 降为 5ms。
- `crypto.dart`：恢复 `streamingThrottleDelayMs=5` 和 `streamingBurstBlocks=16`
- `streaming_decrypt_proxy.dart`：恢复 `blocksSent` + burst 16 块零延迟 + 后续 5ms 节流
- 保留 cipher 复用和跳过冗余 setPosition（这两项优化是正确的）
- 有效供数 ≈ 17MB/s（比原来 10MB/s 快 70%），每块 5ms 让出事件循环

**教训**：`flush()` 不能作为唯一背压——它不等播放器消费。节流延迟不仅限速，还提供事件循环让出点。完全移除节流会同时导致超前解密和事件循环饥饿。

### 二次修正：5ms 仍导致 pipelineFull（恢复 15ms + 降 burst 为 8）

用户反馈：5ms 节流后 seek 移动进度条仍卡顿，日志大量 `pipelineFull: too many frames in pipeline (6)` + `pending=58`。

**根因**：`flush()` 在 127.0.0.1 localhost 回环上几乎零延迟——数据瞬间从 socket 缓冲区进入播放器内部缓冲区，不提供实质背压。5ms 节流仍太快，解密器以 ~17MB/s 灌入，远超播放器消费速率（1080p ~1MB/s），解码管道溢出、帧堆积。seek 后尤其严重——播放器重建解码管道期间无法消费数据。

**修复**：
- `streamingThrottleDelayMs`：5ms → **15ms**（恢复原值，15ms/512KB ≈ 34MB/s 仍远高于播放所需）
- `streamingBurstBlocks`：16 → **8**（4MB），减少 seek 后瞬间灌入量，4MB 足够起播且不溢出管道
- 保留 cipher 复用和跳过冗余 setPosition

**最终教训**：localhost 回环下 `flush()` 无背压，**节流延迟是唯一有效的速率限制**。原版"微卡"的根因是 cipher 重建的 CPU 开销（已由 cipher 复用解决），不是节流太慢。不可盲目降低节流参数。

### 二次修正：5ms 节流仍然太快，pipelineFull 灌爆

用户反馈：5ms 节流后仍卡，特别是 seek 后。日志 `pipelineFull: too many frames in pipeline (6)` + `pending=58`。

**根因**：5ms/512KB ≈ 100MB/s，远超播放器消费速率（1080p ~1MB/s）。localhost 回环下 `flush()` 零延迟（数据瞬间进入播放器内部缓冲区），不提供实质背压。seek 后播放器重建解码管道期间无法消费数据，burst 8MB 瞬间灌入直接溢出管道。

**修复**：
- `streamingThrottleDelayMs`：5ms → **15ms**（恢复原值，≈34MB/s 仍远高于播放所需）
- `streamingBurstBlocks`：16 → **8**（4MB，seek 后更安全，仍够起播）
- 保留 cipher 复用和跳过冗余 setPosition

**核心教训**：localhost HTTP 代理的速率限制**只能靠人为 delay**，`flush()` 在回环上无效。节流值应匹配"播放器消费速率 × 安全倍数"而非"网络带宽"。seek 后是脆弱期，burst 不宜过大。

### 根本性方案：解密移入 worker Isolate（方案 A）

用户反馈：15ms 节流效果差别不大。节流是治标——核心问题是**同步 processBytes（25ms/块）阻塞主 Isolate 事件循环**，导致播放器并发 Range 请求排队。

**架构重构**：把解密完全移出主线程，主线程仅负责 HTTP 响应写入。

**修改文件**：
1. `streaming_decrypt_proxy.dart` — 核心重构
2. `crypto.dart` — 移除 `streamingThrottleDelayMs` 和 `streamingBurstBlocks`（不再需要节流）

**设计**：
- proxy start 时 spawn 长驻 worker Isolate，传 key/iv/encPath（只传一次）
- 每个 Range 请求：主线程发 `decrypt_range` 命令，worker 连续解密回传数据块
- **ack 窗口流控**（替代节流）：worker 每发 2 块后等主线程 ack，主线程 flush 完成才 ack。最多积压 ~1MB，天然防超前解密
- **取消机制**：seek 时旧连接断开，主线程发 `cancel` + `ack` 唤醒 worker 退出
- **天然并发**：worker 的 async 事件循环交替处理多个 decrypt_range（视频轨+音频轨），各请求独立 replyPort + ackReceivePort
- **cipher 复用**：worker 内连续块复用 CTR cipher
- 保留主线程侧 LRU 块缓存（命中时直接返回不经过 worker）

**关键实现点**：
- worker 入口 `_decryptWorkerEntry` 是顶层函数（Isolate.spawn 要求）
- `dart:isolate` 导入
- `ackReceivePort.first` 用于等 ack（每次创建新 subscription，收到事件后 cancel，ReceivePort 本身不关闭）
- worker async listen 在 await 时让出控制权，可接收 cancel 消息设置标志位
- worker 每块解密后检查 `isCancelled()` 退出
