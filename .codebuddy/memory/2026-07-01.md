## SnPlayer 流式解密代理优化

为 SnPlayer Flutter 项目实现了本地 HTTP 代理流式解密方案，替代原有"全量解密后播放"模式，解决大文件解密等待过长问题。

### 新增文件
- `android/app/src/main/res/xml/network_security_config.xml` — Android 网络安全配置，允许 127.0.0.1/localhost 明文 HTTP
- `lib/services/streaming_decrypt_proxy.dart` — 本地 HTTP 代理核心（HttpServer + Range 解析 + AES-CTR 随机访问解密 + 256KB 块流式响应 + 内存 LRU 块缓存 + 磁盘缓存写入）
- `lib/services/playback_cache_manager.dart` — 磁盘缓存管理（完整性校验 + 路径生成 + 过期清理 + LRU 总量管理）

### 修改文件
- `android/app/src/main/AndroidManifest.xml` — 添加 `android:networkSecurityConfig` 引用
- `lib/config/crypto.dart` — 新增流式解密常量（streamingChunkSize=256KB, streamingBlockSize=4MB, streamingMaxCacheBlocks=16, playCacheMaxSize=500MB, playCacheExpireDays=3 等）
- `lib/services/crypto_service.dart` — 新增 `getEncryptedFileInfo()` 方法 + `EncryptedFileInfo` 数据类
- `lib/screens/video_player_screen.dart` — 三段式播放流程（缓存检查→代理播放→降级全量解密）

### 核心架构
利用 AES-256-CTR 随机访问特性 + video_player 的 networkUrl() + 原生播放器 HTTP Range 支持：
1. 播放器发送 Range 请求 → 代理计算密文偏移 + 调整 IV counter
2. 仅解密请求区间的 256KB 块 → 流式返回 + 写入磁盘缓存
3. 二次播放直接从磁盘缓存加载，零解密等待
4. 代理异常自动降级到原有全量解密路径

### 关键技术点
- CTR 模式 16 字节对齐：`aligned_start = (range_start ~/ 16) * 16`，跳过多余字节
- 复用已有 `CryptoUtils.incrementCounter()` 和 `CryptoUtils.createCtrCipher()`
- 内存块缓存 LRU（64MB 上限），磁盘缓存按偏移写入 + 大小校验
- 零新依赖（使用 dart:io HttpServer）

## 流式代理并发竞态 + Content-Type 修复

用户反馈：个别视频播放失败（`contentIsMalformed`）；暂停后卡死无法继续播放。

### 根因分析
1. **Content-Type 固定 `video/mp4`**：非 MP4 格式（MKV/AVI/FLV 等）被 ExoPlayer 用 MP4 解析器解析，导致 `contentIsMalformed`
2. **共享 `RandomAccessFile` 并发竞态**：ExoPlayer 并发发送多个 Range 请求（文件头 + moov atom），多个请求共享同一个 `_encFile`，在 `await` 点交错执行 `setPosition`，导致读到错误位置的数据 → 解码器卡死

### 修复（`streaming_decrypt_proxy.dart` 重写）
1. **动态 Content-Type**：新增 `_guessContentType()`，从加密文件名提取原始扩展名，映射到对应 MIME 类型（mp4/mkv/avi/mov/flv/wmv/webm/3gp/ts）
2. **每请求独立文件句柄**：`_decryptAndStream()` 内部用 `File(_encPath).open()` 打开独立句柄，`finally` 中关闭，彻底消除并发 `setPosition` 竞态
3. **磁盘缓存写入互斥锁**：新增 `_Mutex` 类（基于 `Completer` 队列），`_writeToCache()` 通过 `synchronized()` 串行化写入，避免缓存文件 `setPosition` 竞态

## 流式代理供数速度优化（解决播放卡顿）

用户反馈：播放 2 秒卡住暂停，点播放又 2 秒卡住，画面卡卡的。

### 根因分析
1. **磁盘缓存写入阻塞响应流**：每 256KB 块解密后 `await _writeToCache()` 串行等待磁盘 I/O，阻塞了给 ExoPlayer 的供数
2. **事件循环让出过于频繁**：每 256KB `Future.delayed(Duration.zero)` 一次，调度开销碎片化吞吐量
3. **处理块太小**：256KB 块循环次数多，系统调用开销大

### 修复
1. **处理块从 256KB → 1MB**：新增 `_decryptBlockSize = 1MB` 常量，减少循环和系统调用次数 4 倍
2. **磁盘缓存写入改为 fire-and-forget**：`unawaited(_writeToCache(...))`，复制数据后异步写入，不阻塞 HTTP 响应流
3. **事件循环让出频率降低**：从每 4MB 改为每 8MB 让出一次
4. **磁盘缓存写入完全移除**：流式传输期间不再写磁盘缓存（fire-and-forget 产生 1-2MB 内存复制 + 事件循环积压，是最大瓶颈），`_writeToCache`/`_cacheMutex`/`_Mutex` 类全部移除
5. **块大小 1MB→2MB**：减少循环和系统调用次数

## 流式代理供数吞吐优化

用户反馈：播放 2 秒卡住，点播放又 2 秒卡，画面卡卡。

### 根因
1. **磁盘缓存写入阻塞响应**：每个 256KB 块 `await _writeToCache()`，磁盘 I/O 串行阻塞 HTTP 响应，供数断续
2. **`Future.delayed(Duration.zero)` 让出太频繁**：每 256KB 让出一次事件循环，调度开销大
3. **处理块太小**：256KB 块循环次数多，吞吐碎片化

1. **处理块从 256KB 增大到 1MB**（`_decryptBlockSize`）：减少循环和让出次数
2. **磁盘缓存写入改为 fire-and-forget**：`unawaited(_writeToCache(...))`，复制数据后异步写入，不阻塞 HTTP 响应流
3. **让出频率从每 256KB 改为每 4MB**：`if ((contentLength - remaining) % (4MB) < _decryptBlockSize)`

## 播放频繁 seek 导致卡顿修复

用户反馈：播放卡死，日志显示 `pipelineFull: too many frames (6)` 重复8次、`Render interval 209ms`、5秒仅38帧(7.5fps)、每次触摸触发 MediaCodec FLUSHING→STARTED 循环。

### 根因
加密视频 seek 成本极高（ExoPlayer flush → 新 Range 请求 → 代理重算 IV + 解密 → 重新填充管道）。6 处 `seekTo` 调用全部无节流：进度条拖动每像素、手势每次 drag update 都直接 seek，导致解码器反复重启管道堆积。

### 修复（4 个文件）
1. **player_progress_bar.dart** — 拖动改为松手才 seek：
   - `_onDragStart`/`_onDragUpdate`：仅更新 `_dragValue`（UI 预览），移除 `widget.onChanged` 调用
   - `_onDragEnd`：调用一次 `widget.onSeek`
   - `_onTapDown`：保留立即 seek，移除误设 `_isDragging` 逻辑
   - 移除未使用的 `onChanged` 字段
2. **player_gesture.dart** — 手势拖动改为松手才 seek + 预览浮层：
   - 新增 `_isDragSeeking`/`_dragTargetPosition` 字段，拖动期间只更新预览不 seek
   - `_onHorizontalDragEnd`/`_onVerticalDragEnd`：松手调用一次 `seekTo`
   - build 新增居中时间预览浮层（`_formatDuration`）
   - `_ensurePlayAfterSeek` 改为 `Timer? _playAfterSeekTimer` 防重入
   - 清理未使用的 `_dragStartPosition`/`_verticalDragStartOffset` 字段
   - 修复 `withOpacity` → `withValues`、`final` → `const`
3. **player_controls.dart** — Timer 防重入：
   - `_ensurePlayAfterSeek` 和 play 按钮延迟重试均改为 `Timer?` 字段管理，每次调用取消上一个
   - `dispose` 中取消 Timer
4. **streaming_decrypt_proxy.dart** — `_initialBurstSize` 从 16MB 降为 4MB，避免 seek 后瞬间灌满解码器管道
