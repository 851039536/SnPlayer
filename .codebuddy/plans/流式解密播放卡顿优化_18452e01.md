## 用户需求

播放加密视频时出现卡顿和掉帧现象，特别是大文件（700MB+）尤为明显。需要优化流式解密供数性能，使播放流畅。

## 产品概述

SnPlayer 是一个 Flutter 加密视频播放器，使用 AES-256-CTR 加密 .enc 格式视频文件。播放时通过本地 HTTP 代理服务器（StreamingDecryptProxy）按需 Range 解密，将解密数据流式传输给原生播放器。当前实现存在多处性能瓶颈，导致大文件播放时供数速率不足，引发掉帧。

## 核心问题

- 每个解密块（512KB）都重建 cipher 对象（700MB 文件约 1400 次重建）
- 每块解密后人为延迟 15ms 节流，有效供数速率仅约 10MB/s
- 每块执行 flush + 内存拷贝，增加 GC 压力和上下文切换

## 技术栈

- 框架: Flutter + Dart
- 加密库: PointyCastle (AES-256-CTR)
- 播放器: video_player 包（底层 ExoPlayer / AVPlayer）
- 架构: 本地 HttpServer 代理 + Range 请求按需解密 + LRU 内存块缓存

## 实现方案

### 核心优化策略

**优化 1：Cipher 对象复用（最大收益）**

CTR 模式的 StreamCipher 内部 counter 在 `processBytes` 时自动递增。当前代码每个 512KB 块都调用 `incrementCounter` + `createCtrCipher` 重建 cipher，700MB 文件产生约 1400 次对象创建+GC。

优化方案：在 `_decryptAndStream` 方法内创建局部 `cipher` 和 `cipherPos` 变量，首次创建后跟踪 cipher 当前位置。仅当 `cipherPos != alignedStart`（位置不连续，如 seek 跳转）时才重建 cipher。经分析，同一请求内顺序播放时所有块的位置天然连续（`bytesRead` 恒为 16 字节对齐），cipher 可贯穿整个请求复用，消除 99.9% 的对象创建。

**优化 2：移除人为节流延迟**

当前代码在 burst 16 块后每块延迟 15ms（`streamingThrottleDelayMs`），有效吞吐约 512KB / (25ms解密 + flush + 15ms) ≈ 10MB/s。`response.flush()` 已提供 TCP 背压（socket 缓冲区满时 flush 自动阻塞），15ms 延迟是冗余的双重保险，反而限制了供数速率。

优化方案：移除 `blocksSent` 计数器、`throttleDelay` 计算和 `Future.delayed(throttleDelay)` 调用，仅保留 `flush()` 作为唯一背压机制。预计供数速率提升至 20MB/s+。

**优化 3：跳过冗余 setPosition**

顺序播放时文件句柄位置自然前进，当前每块都调用 `encFile.setPosition(cipherFileOffset)` 是冗余的系统调用。跟踪 `lastFilePos`，仅当位置不匹配时才 setPosition。

### 性能分析

- **时间复杂度**: 解密本身仍为 O(n)（n=文件大小），但常数因子大幅降低（消除对象创建/GC/setPosition 开销）
- **吞吐量**: 预计从 ~10MB/s 提升至 ~20-25MB/s（2-2.5x），远高于视频播放所需 ~1-5MB/s
- **内存**: 无新增内存占用，反而减少 GC 压力（消除 cipher 对象创建）
- **瓶颈**: `processBytes` 同步阻塞 ~25ms/块仍在主 Isolate，如优化后仍卡顿可后续考虑 Isolate 解密

### 关键决策与权衡

1. **保留 512KB 块大小**: 不增大到 1MB/2MB，避免单块同步阻塞时间翻倍影响 UI 响应
2. **保留 flush()**: 作为唯一 TCP 背压，防止内存爆炸；不批量 flush 以保证供数实时性
3. **不引入 Isolate 解密**: 当前优化已足够；Isolate 方案增加复杂度且通信开销可能抵消收益，留作后续备选
4. **保留内存拷贝**: `Uint8List.fromList` 虽有拷贝开销但保证安全性（procBuf 下轮被覆盖），双缓冲方案复杂度高收益低

## 实现注意事项

- CTR cipher 复用正确性：经分析，同一请求内顺序播放时 `bytesRead` 恒为 16 字节对齐（512KB = 32768 × 16），`cipherPos` 与下一块的 `alignedStart` 天然相等，复用安全
- 并发安全：每个 HTTP 请求独立调用 `_decryptAndStream`，局部 `cipher`/`cipherPos` 变量无竞态
- 缓存命中路径不受影响：cipher 复用仅作用于解密路径，cache hit 路径不变
- 向后兼容：不修改加密格式、文件头结构、缓存策略，仅优化供数性能
