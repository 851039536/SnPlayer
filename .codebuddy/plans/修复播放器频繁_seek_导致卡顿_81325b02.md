## 问题概述

加密视频播放时出现严重卡顿，日志显示解码器管道反复溢出（`pipelineFull: too many frames in pipeline (6)` 连续 8 次），5 秒内仅渲染 38 帧（约 7.5fps），每次用户触摸屏幕都触发解码器 FLUSHING→STARTED 循环。

## 根因分析

加密视频通过本地 HTTP 代理流式解密播放，每次 seek 的代价极高（ExoPlayer flush 解码器 → 发起新 Range 请求 → 代理重开加密文件 + 重算 IV + 解密 → 重新填充管道）。当前代码中 6 处 `seekTo` 调用全部无节流，拖动进度条和手势滑动时每像素移动都触发一次完整 seek flush，导致解码器反复重启。

## 核心修复

1. 进度条拖动：改为"松手才 seek"，拖动期间仅 UI 预览
2. 手势拖动：改为"松手才 seek"，拖动期间显示目标时间预览浮层
3. `_ensurePlayAfterSeek` 防重入：用 Timer 字段取消上一次延迟回调
4. 代理 burst 大小优化：seek 后初始冲刷从 16MB 降为 4MB，避免瞬间灌满解码器管道

## 技术栈

- Flutter + Dart（现有项目）
- `video_player` 插件（ExoPlayer 后端）
- 自建流式解密代理（`StreamingDecryptProxy`，本地 HttpServer + AES-256-CTR 按需解密）

## 实现方案

### 修复 1（核心）：进度条拖动改为松手才 seek

**文件**：`lib/widgets/player/player_progress_bar.dart` — `_BufferedSliderState`

当前问题：

- `_onDragStart`（239行）调用 `widget.onChanged` → `controller.seekTo`，拖动开始就 seek
- `_onDragUpdate`（251行）调用 `widget.onChanged` → `controller.seekTo`，每像素移动都 seek
- `_onTapDown`（223行）调用 `widget.onSeek` → `controller.seekTo`，但额外误设 `_isDragging = true`（100ms 后复位），逻辑混乱

修改方案：

- `_onDragStart`：仅设置 `_isDragging = true` + `_dragValue = fraction`，**移除** `widget.onChanged` 调用
- `_onDragUpdate`：仅 `setState(() => _dragValue = fraction)` 更新 UI 预览，**移除** `widget.onChanged` 调用
- `_onDragEnd`：`setState(() => _isDragging = false)` + 调用一次 `widget.onSeek(Duration(milliseconds: seekMs))`
- `_onTapDown`：保留立即 `widget.onSeek`（单击场景合理），**移除** `_isDragging` 的误设逻辑

### 修复 2（核心）：手势拖动改为松手才 seek + 预览浮层

**文件**：`lib/widgets/player/player_gesture.dart` — `_PlayerGestureState`

当前问题：

- `_onHorizontalDragUpdate`（130行）：每次 update 直接 `widget.controller.seekTo`
- `_onVerticalDragUpdate`（151行）：每次 update 直接 `widget.controller.seekTo`

修改方案：

- 新增字段：`Duration? _dragTargetPosition`（拖动期间目标位置）、`bool _isDragSeeking`（是否正在拖动 seek）、`String _dragPreviewText`（预览文本）
- `_onHorizontalDragUpdate`：计算目标位置 → 更新 `_dragTargetPosition` + `_dragPreviewText` → `setState` 刷新预览浮层，**不调用 seekTo**
- `_onHorizontalDragEnd`：调用一次 `widget.controller.seekTo(_dragTargetPosition)` → 清除拖动状态 → `_ensurePlayAfterSeek()`
- 垂直拖动同理
- 预览浮层：在现有 Stack 中新增一个居中的时间提示（如 "→ 12:34"），仅在 `_isDragSeeking` 时显示
- 双击 ±10s（91行）和按钮 ±10s 保留立即 seek（单次操作合理）

### 修复 3：_ensurePlayAfterSeek 防重入

**文件**：`lib/widgets/player/player_gesture.dart` + `lib/widgets/player/player_controls.dart`

当前问题：`_ensurePlayAfterSeek` 使用 `Future.delayed(300ms)` 无取消机制，频繁操作时大量延迟回调堆积，反复调用 `play()`。

修改方案：

- 新增 `Timer? _playAfterSeekTimer` 字段
- `_ensurePlayAfterSeek` 中：先 `_playAfterSeekTimer?.cancel()` 再创建新 Timer
- `dispose` 中取消 `_playAfterSeekTimer`
- `player_controls.dart` 中 play 按钮的延迟重试（181行）同样改为 Timer 字段管理

### 修复 4（优化）：代理 seek 后 burst 大小

**文件**：`lib/services/streaming_decrypt_proxy.dart`

当前 `_initialBurstSize = 16MB`，seek 后新 Range 请求时代理瞬间输出 16MB 不让出事件循环，ExoPlayer 快速读入后灌满解码器管道导致 `pipelineFull`。

修改方案：

- `_initialBurstSize` 从 `16 * 1024 * 1024` 改为 `4 * 1024 * 1024`（4MB）
- 起播场景仍够用（4MB 可覆盖首帧 + moov atom），seek 后不会瞬间灌满管道
- `_yieldInterval` 保持 16MB 不变（稳态供数无问题）

## 实现备注

- 不改动加密/解密流程、不改 `crypto_service.dart`、不改 `crypto.dart` 常量
- 进度条和手势的单击/双击/按钮 seek 保留立即执行（单次操作成本可控）
- 代理核心解密逻辑不变，仅调参数
- 遵循项目代码规范：if 语句必须有花括号 `{}`
